## Project Overview

Soverstack Runtime is a CLI for sovereign infrastructure orchestration. It generates and validates YAML configurations for deploying infrastructure on Proxmox using Terraform and Ansible via Docker.

## Commands

```bash
# Development
npm run dev                    # Run CLI in development (ts-node)
npm run build                  # Compile TypeScript to dist/
npm test                       # Run Jest tests
npm run lint                   # ESLint check
npm run format                 # Prettier format

# CLI commands
soverstack init <project>      # Initialize new project
soverstack validate <path>     # Validate platform.yaml
soverstack plan <path>         # Generate execution plan
soverstack apply <path>        # Apply infrastructure
soverstack destroy <path>      # Destroy infrastructure
```

## Project Structure Generated by Init

### Mode 1: Single Datacenter (default)

```
project/
├── platform.yaml           # Root config - references all layers
├── datacenter.yaml         # Physical servers (Proxmox nodes)
├── networking.yaml         # Public IP, DNS, VPN, Firewall
├── security.yaml           # Vault, SSO (Keycloak/Authentik)
├── observability.yaml      # Monitoring, logging
├── ssh_config.yaml         # SSH users/keys
├── .env                    # Environment variables (never commit!)
├── compute/
│   └── core-compute.yaml   # VM definitions
├── database/
│   └── core-database.yaml  # PostgreSQL clusters
└── .soverstack/
    ├── state/              # Terraform state
    ├── logs/               # Execution logs
    └── cache/              # Temporary files
```

### Mode 2: Multi-Datacenter (`--dc paris,frankfurt`)

```
project/
├── platform.yaml           # References all DCs
├── .env                    # Global env vars
└── datacenters/
    ├── paris/
    │   ├── datacenter.yaml
    │   ├── networking.yaml
    │   ├── security.yaml
    │   ├── observability.yaml
    │   ├── ssh_config.yaml
    │   ├── .env
    │   ├── compute/
    │   │   └── core-compute.yaml
    │   └── database/
    │       └── core-database.yaml
    └── frankfurt/
        └── ... (same structure)
```

**Note:** Users can add a `shared/` folder manually if needed for shared configs across datacenters.

### Environments (dev/staging/prod)

Soverstack does **not** generate separate environment folders to keep the structure simple. Users who need multiple environments have two options:

**Option A** - Use datacenters as environments:
```bash
soverstack init my-project --dc prod,staging,dev
```
This creates `datacenters/prod/`, `datacenters/staging/`, `datacenters/dev/`.

**Option B** - Create separate projects:
```bash
soverstack init my-app-prod --tier production
soverstack init my-app-staging --tier local
```

This avoids deep nesting like `environments/prod/datacenters/paris/compute/` which becomes hard to navigate.

## Architecture

### Type Separation

- `src/types.ts` - Type definitions only (interfaces, type aliases)
- `src/constants.ts` - Runtime constants (`VM_ID_RANGES`, `HA_REQUIREMENTS`)

### Key Types (src/types.ts)

```typescript
// Root configuration
interface Platform {
  project_name: string;
  version: string;
  domain: string;
  infrastructure_tier: InfrastructureTierType;
  layers: {
    datacenter: string;
    compute: string;
    database?: string;
    networking?: string;
    security?: string;
    observability?: string;
  };
  state: { backend: BackendType; path: string };
}

// Infrastructure tier
type InfrastructureTierType = "local" | "production" | "enterprise";

// Credentials (NEVER plain text!)
type CredentialRef =
  | { type: "vault"; path: string }
  | { type: "env"; var_name: string }
  | { type: "file"; path: string };

// Validation output
interface ValidationResult {
  valid: boolean;
  errors: ValidationError[];
  warnings: ValidationWarning[];
}
```

### Validation Pipeline

`src/commands/validate/logic.ts` orchestrates:
1. Load `platform.yaml`
2. Normalize infrastructure (`normalizer.ts` - merges all layer files)
3. Apply defaults (`defaults.ts`)
4. Validate each layer: Datacenter → Networking → Compute → Database → Cluster

### Multi-File Merge

Layers like `compute` and `database` support comma-separated paths:
```yaml
layers:
  compute: "compute/core-compute.yaml,compute/apps-compute.yaml"
```
Arrays are concatenated; duplicate non-array properties cause errors.
