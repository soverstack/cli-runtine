# CLAUDE.md

This file provides guidance to Claude Code when working with this repository.

## Project Overview

Soverstack Runtime is a CLI for sovereign infrastructure orchestration. It generates YAML configurations and Ansible artifacts for deploying infrastructure on Proxmox bare metal servers. The CLI runs inside a Docker image alongside Ansible and Terraform.

## Commands

```bash
# Development
npm run dev                    # Run CLI via ts-node
npm run build                  # Compile TypeScript to dist/
npm test                       # Run Jest tests (81 tests, 4 suites)
npm run lint                   # ESLint check
npm run format                 # Prettier format

# CLI commands
soverstack init [project]              # Initialize new project
soverstack validate [path]             # Validate project configuration
soverstack plan [path] [-v] [--debug]  # Show execution plan (diff desired vs state)
soverstack apply [path] [-v] [--debug] # Apply infrastructure changes
soverstack generate ssh [--all|--region|--dc]  # Generate/rotate SSH keys
soverstack add region                  # Add a region to existing project
soverstack add zone                    # Add a zone to existing project
```

### Key Concepts

- **Region**: Geographic area (eu, us, asia). Has zones and optionally a hub.
- **Hub**: Backup/storage datacenter (HDD, MinIO, PBS). prefix: `hub-*`
- **Zone**: Production compute datacenter (NVMe, Ceph). prefix: `zone-*`
- **Shared hub**: A region can reference another region's hub (`hub: hub-eu` in asia/region.yaml). No files are duplicated.
- **Control plane**: The zone running global services. Set in `platform.yaml` → `defaults.global_placement.datacenter`.
- **Tier**: `local` (1 node, no hub), `production` (3+ nodes, hub required), `enterprise` (5+ nodes, backup enforced).
- **Datacenters are discovered from filesystem** (hub-*/zone-* directories), NOT from region.yaml. region.yaml has no `datacenters:` array.

### VM ID Scheme

Hierarchical, deterministic, collision-free. See `src/commands/init/VM_ID_SCHEME.md`.

```
vm_id = SCOPE_BASE + regionId × 100000 + dcId × 1000 + ROLE_OFFSET + instance

GLOBAL:    1000 + roleOffset + instance              (1000-1499)
REGIONAL:  regionId × 100000 + roleOffset + instance (dcId = 0)
ZONAL:     regionId × 100000 + dcId × 1000 + roleOffset + instance

Example: 102051 → region 1 (eu), dc 2 (zone-paris), offset 50 (loadbalancer), instance 1
```

- `vmId()` function in `init/types.ts` computes all IDs
- `ROLE_OFFSETS` defines offsets per role (step of 50)
- `regionIds` and `dcIds` maps are built in `ProjectInitializer` constructor
- Regions are 1-indexed, dcId 0 = regional services, dcId 1+ = datacenters (hubs first, then zones)

### Implementations & Versions (Single Source of Truth)

All in `src/commands/init/types.ts`:
- `ImplementationMap` — TypeScript type (compile-time)
- `IMPLEMENTATIONS` — Runtime map: role → string[] (used by validators, generators)
- `DEFAULT_IMPLEMENTATIONS` — Default impl per role
- `VERSION_CATALOG` — Current + supported versions per implementation
- `implLine(role)` — Generates YAML: `implementation: postgresql    # postgresql | mysql | mariadb`
- `versionLine(impl)` — Generates YAML: `version: "16"              # Supported: 16, 15, 14`

### Validation (2 layers)

1. **Zod schemas** (`validate/schemas/`) — Structure, types, formats, enums
2. **Custom validators** (`validate/validators/`) — Cross-file references, uniqueness, HA constraints, SSH key age

Both run in `validate/logic.ts`. Zod catches structural issues, custom validators catch logic issues.

Error messages are human-readable (no array indices like `services[0].instances[0]`). Technical field paths only shown with `-v`.

### Plan & Apply

- `plan` reads desired state (YAML) and current state (state.json), computes diff
- `apply` generates Ansible artifacts in `.soverstack/ansible/`, executes phases in order
- State is updated after each successful action (not just at the end)
- Deploy order: Bootstrap → Global (DB → Vault → Identity → DNS → Mesh) → Regional → Zonal

### SSH Key Management

- Minimum 2 users required (admin + backup) for safe rotation
- `generate ssh` backs up keys to `.ssh/.previous/` before regenerating (additive, per-DC)
- If `.previous/` already has keys for a DC, generation is blocked until `apply` deploys them
- `apply` cleans `.previous/` per-DC after successful rotation
- No `--force` flag — the only way to deploy pending keys is `apply`
- Key file content is hashed (sha256) in node config hash for change detection
- `rotation_policy` in ssh.yaml: `validate` warns on approaching/expired keys (never blocks)
- ssh.yaml header warns: "NEVER edit SSH key files manually. Always use: soverstack generate ssh"

### Credential Types

```yaml
# 3 types, used for bootstrap passwords and SSH keys
password:
  type: env          # from .env via --env-file
  var_name: PVE_01_PW

private_key:
  type: file         # from .ssh/ via volume mount
  path: .ssh/zone-paris_admin

private_key:
  type: vault        # from HashiCorp Vault (post-deploy)
  path: secret/soverstack/ssh/zone-paris/admin
```

Phase 1 (bootstrap): `env` + `file`. Phase 3 (post-Vault): `vault`. Migration via future `secrets push` command.

### Docker Architecture

```
HOST                              DOCKER (soverstack/runtime)
┌─────────────────┐              ┌──────────────────────────────┐
│  svk (launcher)  │  docker run │  Node.js CLI + Ansible + TF  │
│  thin script     │ ──────────→ │  CLI validates, plans, gens  │
│                  │             │  artifacts, then runs Ansible │
└─────────────────┘              └──────────────────────────────┘
```

Secrets (.env, .ssh/) passed at runtime via `--env-file` and `-v :ro` mounts. Never baked into image.

## Testing

```bash
npm test                       # All tests (81 tests, 4 suites)
npx jest types.test.ts         # Single file
npx jest -t "vmId"             # Single test by name
npm run test:watch             # Watch mode
```

Test files are in `__tests__/` directories next to the code they test:
- `init/__tests__/types.test.ts` — VM ID scheme, implementations, helpers
- `init/__tests__/generators.test.ts` — Project structure, YAML content, VM ID uniqueness, shared hub
- `validate/__tests__/schemas.test.ts` — Zod schema validation for all file types
- `plan/__tests__/planner.test.ts` — State management, plan computation, deploy order
